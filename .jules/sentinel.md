## 2024-06-18 - XSS Risk with `dangerouslySetInnerHTML` for Email Previews
**Vulnerability:** A cross-site scripting (XSS) risk existed where sanitized HTML for an email preview was rendered into the main DOM using `dangerouslySetInnerHTML`.
**Learning:** Even when using sanitizers like `DOMPurify`, rendering untrusted user-generated content (like email bodies) directly into the main DOM with `dangerouslySetInnerHTML` exposes the application to potential bypasses and disrupts document isolation.
**Prevention:** To prevent XSS when rendering dynamic or untrusted HTML (e.g., email previews), prioritize using a sandboxed `<iframe>` with the `srcDoc` attribute instead of `dangerouslySetInnerHTML`, even when used with `DOMPurify`. Use `sandbox="allow-popups allow-popups-to-escape-sandbox"` to permit link clicking while omitting `allow-scripts` and `allow-same-origin`. Because an iframe creates an isolated browsing context, applying CSS classes (like Tailwind) directly to the `<iframe>` tag will not style the inner document; you must manually inject an inline `<style>` tag into the `srcDoc` HTML string to apply fonts, colors, or sizing to the content itself.
