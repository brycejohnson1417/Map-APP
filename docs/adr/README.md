# Architecture Decision Records

## Purpose

ADRs record durable architecture choices that affect more than one feature or tenant. They are not task notes.

Each ADR should include:

- status
- context
- decision
- rationale
- alternatives considered
- consequences
- follow-up checks

If a decision changes, update the ADR or add a superseding ADR. Do not leave conflicting architecture rules split across product docs and ADRs.
